# 웹 접근성 코딩 규칙 (KWCAG 2.1 준수)

이 프로젝트의 모든 화면·컴포넌트는 **한국형 웹 콘텐츠 접근성 지침 2.1(KWCAG 2.1)** 를 준수한다.
KWCAG 2.1은 **4개 원칙 · 13개 지침 · 24개 검사항목**으로 구성된다. 아래 규칙은 24개 검사항목 **전부**를 Next.js(App Router) / React / TailwindCSS 코딩 관점으로 옮긴 것이며, 각 규칙 앞의 `[번호]` 는 대응하는 공식 검사항목이다.

> - 코드를 작성·수정할 때 아래 규칙을 항상 적용한다. 새 컴포넌트는 마지막의 **PR 체크리스트**를 통과해야 한다.
> - ⚠️ **주의**: 이 문서를 지키는 것은 인증의 *필요조건*이지 *충분조건*이 아니다. 웹 접근성 품질인증마크는 인증기관의 실물 심사(자동+전문가 수동+장애인 사용성 평가, 표본 페이지)로 발급되며, 실제 콘텐츠 품질(대체 텍스트의 적절성, 실제 자막·색상값 등)까지 평가한다. 문서에서 검증할 수 없는 항목은 `📄 콘텐츠 검수 필요`로 표시한다.
> - 본 프로젝트의 준수 기준은 **KWCAG 2.1**이다(프로젝트 요구사항). 상위 개정판 2.2가 존재하나 본 프로젝트 범위는 아니다.

## 원칙 1. 인식의 용이성 (Perceivable) — 검사항목 7

### 지침 5.1 대체 텍스트

- **[5.1.1] 적절한 대체 텍스트 제공** — 의미 있는 이미지·아이콘·차트에 대체 텍스트를 제공한다.
    - `next/image`의 `<Image>` 및 `<img>`에는 반드시 `alt`를 명시한다. 장식용은 `alt=""`(빈 문자열)로 두고, 아예 생략하지 않는다.
    - 아이콘만으로 된 버튼/링크에는 `aria-label` 또는 `sr-only` 텍스트를 제공하고, 내부 아이콘은 `aria-hidden="true"`.
    - 📄 콘텐츠 검수 필요: 대체 텍스트가 이미지의 *의미*를 정확히 전달하는지(파일명·"이미지" 등 부적절 금지).
    ```tsx
    <button aria-label="메뉴 열기"><MenuIcon aria-hidden="true" /></button>
    <Image src="/logo.png" alt="회사 로고" width={120} height={40} />
    <Image src="/deco.png" alt="" width={200} height={100} /> {/* 장식용 */}
    ```

### 지침 5.2 멀티미디어 대체수단

- **[5.2.1] 자막 제공** — 동영상에 자막(`<track kind="captions" srcLang="ko">`), 음성에 대본(transcript)이나 수화를 제공한다. 📄 콘텐츠 검수 필요(실제 자막 파일 존재·싱크).

### 지침 5.3 명료성

- **[5.3.1] 색에 무관한 콘텐츠 인식** — 색만으로 정보를 전달하지 않는다(필수표시·상태·그래프 계열). 아이콘·텍스트·패턴을 병행한다.
- **[5.3.2] 명확한 지시사항 제공** — 감각 특성(모양·위치·색·소리)에만 의존한 안내를 금지한다. "오른쪽 초록 버튼", "아래 원형 아이콘" ❌ → 텍스트 레이블/이름을 함께 제시("**저장** 버튼").
- **[5.3.3] 텍스트 콘텐츠의 명도 대비** — 일반 텍스트 **4.5:1 이상**, 큰 텍스트(18.66px bold / 24px↑) **3:1 이상**. 저대비 회색을 본문에 쓰지 않는다. 본문은 시맨틱 토큰 `text-foreground`(기본) · `text-foreground-muted`(대비 확보된 보조 색)만 사용하고, 임의 팔레트(`text-gray-400` 등) 하드코딩은 피한다([NA-009] 연계).
- **[5.3.4] 자동 재생 금지** — 3초 이상 자동재생 소리를 금지한다. 자동재생 미디어에는 정지/음소거 컨트롤을 둔다. `<video autoPlay>` 지양, 부득이하면 `muted` + 컨트롤 제공.
- **[5.3.5] 콘텐츠 간의 구분** — 이웃한 콘텐츠·UI 컴포넌트·경계선은 인접 색과 **3:1 이상** 대비 또는 테두리/간격으로 구분한다.

## 원칙 2. 운용의 용이성 (Operable) — 검사항목 9

### 지침 6.1 입력장치 접근성

- **[6.1.1] 키보드 사용 보장** — 모든 기능을 키보드만으로 사용할 수 있어야 한다. 클릭 동작은 `<button>`/`<a>`로 구현하고 `<div onClick>`을 쓰지 않는다. 커스텀 위젯은 `role`, `tabIndex={0}`, `onKeyDown`(Enter/Space) 처리를 추가한다.
- **[6.1.2] 초점 이동** — 포커스 순서가 논리적이고, 포커스 표시를 제거하지 않는다. `outline-none` 단독 금지 → 항상 `focus-visible:` 대체 스타일 동반. 양수 `tabindex` 사용 금지.
    ```tsx
    className = 'focus:outline-none focus-visible:ring-2 focus-visible:ring-brand focus-visible:ring-offset-2'
    ```
- **[6.1.3] 조작 가능** — 클릭·터치 대상은 최소 **44×44px** 이상(`min-h-11 min-w-11` 또는 충분한 `p-*`). 인접 타깃 간 간격 확보.

### 지침 6.2 충분한 시간 제공

- **[6.2.1] 응답시간 조절** — 시간 제한(세션 만료 등)은 **연장·해제·조절** 수단을 제공한다. 예: 만료 전 경고 + "연장" 버튼.
- **[6.2.2] 정지 기능 제공** — 자동으로 움직이거나 갱신되는 콘텐츠(캐러셀·자동 슬라이드·스크롤 배너)에 **정지 버튼**을 제공한다.

### 지침 6.3 광과민성 발작 예방

- **[6.3.1] 깜빡임과 번쩍임 사용 제한** — 1초에 **3~50회** 범위의 번쩍임을 만들지 않는다. `prefers-reduced-motion`을 존중한다.
    ```tsx
    className = 'motion-safe:animate-fade-in motion-reduce:animate-none'
    ```

### 지침 6.4 쉬운 내비게이션

- **[6.4.1] 반복 영역 건너뛰기** — 레이아웃 최상단에 본문 바로가기(Skip Link)를 둔다.
    ```tsx
    <a href="#main" className="sr-only focus:not-sr-only focus:absolute focus:top-2 focus:left-2">본문 바로가기</a>
    <main id="main">…</main>
    ```
- **[6.4.2] 제목 제공** — 라우트별 `export const metadata = { title }`를 지정한다. `<h1>`은 페이지당 하나, 헤딩 레벨을 건너뛰지 않는다(크기는 Tailwind 클래스로 조절, 헤딩 레벨 오용 금지). 시맨틱 랜드마크(`<header> <nav> <main> <footer>`) 사용.
- **[6.4.3] 적절한 링크 텍스트** — 링크 텍스트만으로 목적을 알 수 있게 한다. "여기 클릭"·"더보기" 단독 금지 → 맥락 포함 또는 `aria-label` 보완.

## 원칙 3. 이해의 용이성 (Understandable) — 검사항목 6

### 지침 7.1 가독성

- **[7.1.1] 기본 언어 표시** — 루트 레이아웃의 `<html lang="ko">`를 유지한다. 부분적으로 다른 언어 구간은 해당 요소에 `lang`을 지정한다.

### 지침 7.2 예측 가능성

- **[7.2.1] 사용자 요구에 따른 실행** — 포커스·입력만으로 의도치 않은 큰 변화(자동 이동·팝업·제출)를 일으키지 않는다. 실행은 명시적 버튼/링크로. `onChange`에서 곧바로 페이지 이동/제출하지 않는다.

### 지침 7.3 콘텐츠의 논리성

- **[7.3.1] 콘텐츠의 선형 구조** — DOM 순서 = 읽기 순서가 되도록 마크업한다. 시각적 위치는 CSS로만 바꾸고, 의미 순서를 깨는 배치(예: `order`, `flex-row-reverse`, 절대배치)로 논리 순서를 뒤엎지 않는다.
- **[7.3.2] 표의 구성** — 데이터 표는 `<caption>`, `<th scope="col|row">`로 구조화한다. 레이아웃 목적으로 `<table>`을 쓰지 않는다(Flex/Grid 사용). 복잡한 표는 `headers`/`id`로 셀을 연결한다.

### 지침 7.4 입력 도움

- **[7.4.1] 레이블 제공** — 모든 폼 입력에 레이블을 연결한다. `<label htmlFor>` ↔ `id`, 불가피하면 `aria-label`/`aria-labelledby`. placeholder는 레이블을 대체하지 못한다.
    ```tsx
    <label htmlFor="email">이메일</label>
    <input id="email" type="email" autoComplete="email" required aria-required="true"
           aria-invalid={hasError} aria-describedby={hasError ? "email-err" : undefined} />
    {hasError && <p id="email-err" role="alert">올바른 이메일을 입력하세요.</p>}
    ```
- **[7.4.2] 오류 정정** — 입력 오류를 명확히 안내하고 정정 방법을 제시한다. 오류 필드에 `aria-invalid="true"`, 메시지에 `id`를 부여해 `aria-describedby`로 연결, 요약/메시지는 `role="alert"`(또는 `aria-live="assertive"`)로 즉시 전달.

## 원칙 4. 견고성 (Robust) — 검사항목 2

### 지침 8.1 문법 준수

- **[8.1.1] 마크업 오류 방지** — 유효한 마크업: 요소 중첩 규칙 준수, `id` 중복 금지, 속성 올바르게 사용, 여닫는 태그 짝 맞춤. 상호작용 요소는 올바른 태그로(링크 `<a>`, 버튼 `<button>`).

### 지침 8.2 웹 애플리케이션 접근성

- **[8.2.1] 웹 애플리케이션 접근성 준수** — 표준 시맨틱 태그로 충분하면 ARIA보다 우선한다("No ARIA is better than bad ARIA"). 커스텀 위젯(모달·탭·아코디언·드롭다운)에는 올바른 `role`과 상태(`aria-expanded`·`aria-selected`·`aria-controls`·`aria-modal` 등) + 키보드 상호작용을 구현한다. 모달은 **포커스 트랩** + 닫은 뒤 **포커스 복귀**, 배경 `inert`/`aria-hidden`. 동적 갱신 영역(토스트·로딩)은 `aria-live`로 알린다.

## 검증 방법

- **자동 점검**: `eslint-plugin-jsx-a11y`(정적 분석), axe DevTools / Lighthouse Accessibility 감사, HTML validator(8.1.1).
- **수동 점검**: 키보드만으로 전체 흐름 조작, 스크린리더(NVDA·VoiceOver) 확인, 확대 200%·명도 대비 측정.
- 자동 도구는 접근성 문제의 약 30~40%만 잡으므로 **수동 점검을 반드시 병행**한다.

## 검사항목 커버리지 매핑

전체 24개 검사항목이 위 규칙에 1:1로 대응한다(각 규칙의 `[번호]` 참조).
단, 다음은 **코드가 아니라 실제 콘텐츠·운영에서 확인**해야 한다(문서만으로 보증 불가):
`5.1.1`(대체 텍스트 적절성) · `5.2.1`(실제 자막) · `5.3.3`(실제 색상값) · 그리고 표본 페이지 전수 일관성.

## 인증마크(웹 접근성 품질인증) 취득 시 유의

- 발급: 과기정통부 지정 인증기관 심사 → 유효기간 **1년**, 갱신 필요.
- 평가: 자동 진단 + 전문가 수동 평가 + **장애인 당사자 사용성 평가**, 사이트에서 **표본 페이지 추출**.
- 이 문서는 개발 단계 준수를 보장할 뿐, 최종 취득은 위 심사 통과가 필요하다.
- 참고: 인증마크 심사 기준 버전은 시기에 따라 다를 수 있으므로, **인증 취득이 목표라면 적용 버전(2.1/2.2)을 발주처·인증기관과 사전에 확인**한다. 본 프로젝트의 적용 기준은 **2.1**이다.
